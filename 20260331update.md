# 2026-03-31 Update: Fix SD Card Logging — Empty Folders

## Problem

After flashing both boards, the system appeared to work:
- STM32 I2C scan found the Grove AI at **0x28** (not the expected 0x62)
- I2C start-recording command was received (Grove AI printed `[I2C_CMD] Recording started`)
- Session folders (`SESSION_XXXX/ALL/`, `SESSION_XXXX/DETECT/`) were created on the SD card

But **no image files were ever written** inside the folders.

## Root Causes Found

### 1. SPI pin mux clobbered by sensor init (primary cause)

`sdlog_session_init()` configured PB2–PB5 for SD card SPI, but was called **before** `app_start_state()`. The sensor/datapath initialization inside `app_start_state()` reconfigured the SPI pin mux, breaking the SD card connection. Folders were created during init (before the pin mux was disturbed), but all subsequent writes in the frame callback failed silently.

### 2. Blocking UART sends consumed entire frame budget

`UART_SEND_ALOGO_RESEULT` was defined in the `.mk`, causing `cv_yolov8n_ob_run()` to base64-encode the full JPEG image and send it over UART every frame (~300ms blocking at 921600 baud). At 4fps (250ms/frame), this alone exceeded the frame period, leaving zero time for SD card writes.

### 3. D-cache invalidation instead of clean+invalidate

`sdlog_write_image()` used `SCB_InvalidateDCache_by_Addr` which only discards cached lines. The correct call is `SCB_CleanInvalidateDCache_by_Addr` which first flushes dirty data back to main memory, then invalidates — ensuring the JPEG data written by DMA is visible to the CPU before writing to SD.

## Changes Made

### Grove Vision AI V2 — `tflm_yolov8_od_sdlog`

**`tflm_yolov8_od_sdlog.c`** — Reordered init sequence:
```
BEFORE:  sdlog_session_init() → i2c_cmd_init() → cv_yolov8n_ob_init() → app_start_state()
AFTER:   i2c_cmd_init() → cv_yolov8n_ob_init() → app_start_state() → sdlog_session_init()
```
- I2C callback registered first (must be in place before events start)
- SD card mounted last (after all sensor/pin mux changes are finalized)
- Added diagnostic print on first logged frame: `[SDLOG] First frame: jpeg_addr=... jpeg_sz=...`

**`tflm_yolov8_od_sdlog.mk`** — Disabled `UART_SEND_ALOGO_RESEULT`:
```makefile
#APPL_DEFINES += -DUART_SEND_ALOGO_RESEULT  # disabled: conflicts with SD logging
```
- Eliminates the ~300ms/frame blocking UART send
- Detection results are still logged to `session.log` on the SD card

**`sdlog.c`** — Three fixes in `sdlog_write_image()`:
- `SCB_InvalidateDCache_by_Addr` → `SCB_CleanInvalidateDCache_by_Addr`
- `FA_CREATE_NEW` → `FA_CREATE_ALWAYS` (won't fail on filename collision)
- Added early-out guard for `addr==0` or `sz==0` with diagnostic print
- Moved cache operation before `f_open` (ensures data ready before any SD IO)

### STM32 — `i2cscan`

**`Core/Src/main.c`** — Fixed misleading I2C address log:
- Was hardcoded `"found at 0x62"` even though the actual probe address was 0x28
- Now prints the actual address from `GROVE_I2C_ADDR`

## I2C Address Note

The Grove AI firmware configures `EVT_I2CS_0_SLV_ADDR = 0x62`, but the device appears on the bus at **0x28**. This is a quirk of the Himax DesignWare I2C slave driver's internal address mapping. The STM32 uses `0x28` and it works — no change needed on either side. The internal callback routing on the Grove side uses the configured value (0x62) and functions correctly.
